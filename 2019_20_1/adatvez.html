Adatvezérelt rendszerek

Bevezetés

  • Adatvezérelt rendszer
    • Nagy adatbázis tartozik hozzá
    • Nyilvántartja az adatokat, amit esetleg felhasználóknak megjelenít
    • Pl.
      • Neptun
      • Facebook
      • airbnb
    • Nem adatvezérelt rendszer pl.:
      • Játékok
      • Tisztán kliens oldali alkalmazások
      • Az adatokat nem tartja nyilván, csak adott sessionben bevitt adatokkal dolgozik
    • Mi kell egy adatvezérelt rendszerhez?
      • Fejlesszünk Gmailt
      • Weboldal, mobil app, POP3 kliens, csatolmányok, többnylevűség stb.
      • Keressünk design patternt, a már létező hasonló alkalmazások alapján
        • Háromrétegű architektúra
        • Ez egy jellemző pattern adatvezérelt rendszereknél
      • Presentation: HTML CSS JS | Android components
      • Services: REST interface
      • Business Layer: basic functionality
      • Data Access Layer: üzenetek, felhasználók, csatolmányok
      • Database: adtabázis, Google Drive csatolmányok
    • Háromrétegű architektúra
      • Prezentation layer
        • Összeköti az üzleti logikát a userrel
        • Kliens oldali alkalmazás
        • Weboldal, applikáció stb.
      • Business logics layer
        • Szerver oldal
        • API jellegű
      • Data layer
        • Data access layer
        • Csatlakozik adatbázishoz
        • Csatlakozik külső szolgáltatásokhoz
        • Egy absztrakciós réteg az adatok fölé
          • Cserélhetővé teszi a szolgáltatásokat és adatbázist
          • Absztrakt utasításokat fogad, melyeket közvetít API hívással, vagy SQL querykkel
      • Kiegészítés: Services layer
        • Többrétegű architektúra
        • Prezentációs és üzleti réteg közé
        • Akár tekinthető a Business layer input output interface deklarációjává
        • Rögzített üzenettípusokká konvertálja a szerver válaszát
        • Rögzíti az input interfacet
      • Cross cutting concernes
        • Rétegen kívüli komponensek
        • Pl.: security issues
          • Autentikáció, autorizáció
        • Naplózás
        • Kommunikáció
          • Rétegek közt
          • Szinkron, aszinkron
          • Hálózati hívások / fv hívások / alkalmazások közti hívások
    • Frontend vs backend
      • Frontend: prezentációs layer
      • Backend: minden ami nem a prezentációs layer
  • Félév szerkezete
    • Adatkapcsolati réteg
      • MS SQL Server, PL/SQL
      • MongoDB
      • Párhuzamos adathozzáférés (tranzakciók)
      • Platform specifikus adatforrás
      • Adatstruktúrák (XML, JSON)
    • Business layer
      • Java, .NET
      • O/R lekérdezések
      • Entity Framework, JPA
      • REST WebAPI

Tranzakció kezelés

  • Tranzakció
    • Olyan logikai lépések, melyek végrehajtása csak együtt értelmes
    • ACID elvek
      • Atomicitás, konzisztencia
        • A köztes állapotokat senki nem láthatja
        • A lépés sorozat kívülről 1 egységes lépésként látszik
      • Izoláció
        • A tranzakció ne tudja, hogy más tranzakció is fut
      • Durability
        • Ami COMMITolt, az úgy is marad
    • Csak az izoláció a probléma
      • Dirty reading, phantom reading, lost modification, nem megismételhető olvasás
      • Megoldás: helyes ütemezés, várakoztatás
        • Helyes ütemezés: minden lekért és módosított rekord olyan, mintha soros ütemezés történt volna
      • 2PL
        • WR lockok
        • Elképzelhető Deadlock
      • 4 izolációs szint
        • Read uncommitted
          • Minden tranzakció azt csinál amit akar
        • Read committed (default)
          • Dirty read megoldódik, de a többi nem
        • Repeatable read
          • Nincs se dirty, se unrepeatable read
        • Serializable
          • Egyik probléma se fordulhat elő
